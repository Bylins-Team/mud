# План работы: Пофайловый анализ безопасности Bylins MUD

## Цель
Проанализировать все 896 файлов кодовой базы на наличие уязвимостей и ошибок.

## Статус
- **Всего файлов**: 896
- **Проанализировано**: см. PROGRESS.txt
- **Текущий файл**: см. PROGRESS.txt (последняя строка)

## Как продолжить работу

### Команда для продолжения:
```
Продолжи работу, спланированную в файле bugs/WORK_PLAN.md
```

### Шаги для продолжения:
1. Прочитать `bugs/PROGRESS.txt` - найти номер последнего проанализированного файла
2. Взять следующий файл из `bugs/all_files.txt` (номер+1)
3. Проанализировать файл по критериям ниже
4. Записать результат в `bugs/findings/XXX_description.md` (если проблемы найдены)
5. Обновить `bugs/PROGRESS.txt`
6. Перейти к следующему файлу

### Пример кода для продолжения:
```bash
# Узнать текущий прогресс
grep "^\[" bugs/PROGRESS.txt | tail -1

# Взять следующий файл
LAST=$(grep -oP '^\[\K[0-9]+' bugs/PROGRESS.txt | tail -1)
NEXT=$((LAST + 1))
sed -n "${NEXT}p" bugs/all_files.txt
```

## Критерии анализа

Для каждого файла проверять:

### 1. Безопасность памяти
- [ ] `strcpy`, `strcat`, `sprintf` без проверки размера → Buffer Overflow
- [ ] `sscanf` с `%s` без ограничения длины → Buffer Overflow
- [ ] Обращение к массиву/строке без проверки границ → Out of Bounds
- [ ] `malloc`/`new` без соответствующего `free`/`delete` → Memory Leak
- [ ] Использование указателя после `free`/`delete` → Use-After-Free
- [ ] Разыменование без проверки на NULL → Null Pointer Dereference

### 2. Инъекции
- [ ] `system()`, `popen()` с пользовательскими данными → Command Injection
- [ ] SQL-запросы со строковой конкатенацией → SQL Injection
- [ ] Использование путей файлов с `../` → Path Traversal
- [ ] Format string с пользовательскими данными → Format String Vulnerability

### 3. Целочисленные переполнения
- [ ] Арифметика с user input без проверки → Integer Overflow/Underflow
- [ ] Использование результата как размера массива → Potential Overflow

### 4. Логические ошибки
- [ ] Race conditions (глобальные переменные без мьютексов)
- [ ] Некорректные проверки условий
- [ ] Off-by-one errors в циклах
- [ ] Неправильная обработка ошибок

### 5. Качество кода
- [ ] Закомментированный код (>10 строк)
- [ ] Неиспользуемые переменные
- [ ] Дублирование кода
- [ ] Нарушение стиля проекта

## Формат записи результатов

### Если проблемы НЕ найдены:
Добавить в PROGRESS.txt:
```
[N/896] путь/к/файлу.cpp - OK (проблем нет)
```

### Если проблемы найдены:
1. Создать файл `bugs/findings/XXX_description.md`:
```markdown
# Краткое описание проблемы

**Файл**: путь/к/файлу.cpp
**Строки**: XX, YY
**Критичность**: CRITICAL/HIGH/MEDIUM/LOW

## Проблема
Описание с примером кода

## Воспроизведение
Как проявляется проблема

## Решение
Как исправить с примером кода
```

2. Добавить в PROGRESS.txt:
```
[N/896] путь/к/файлу.cpp - DONE (X проблем: 004, 005)
```

3. Обновить счётчик в шапке PROGRESS.txt

## Приоритеты критичности

- **CRITICAL**: Позволяет выполнить произвольный код, получить полный контроль
  - Command Injection
  - SQL Injection (если есть)
  - Buffer Overflow с возможностью RCE
  - Privilege Escalation

- **HIGH**: Серьёзная уязвимость, компрометация данных
  - Buffer Overflow (без очевидного RCE)
  - Path Traversal
  - Use-After-Free
  - Information Disclosure (пароли)
  - Race Conditions с security impact

- **MEDIUM**: Умеренная проблема
  - Логические ошибки
  - Null Pointer Dereference
  - Integer Overflow (без явного impact)
  - Memory Leaks (небольшие)

- **LOW**: Качество кода, minor проблемы
  - Закомментированный код
  - Code smell
  - Небольшие memory leaks
  - Нарушения стиля

## Нумерация находок

- Использовать **сквозную нумерацию**: 001, 002, 003, ...
- Номер = текущий максимальный номер + 1
- Проверять: `ls bugs/findings/ | grep -oP '\d+' | sort -n | tail -1`

## Особые случаи

### Header файлы (.h)
Обычно проблем меньше, но проверять:
- Отсутствие include guards
- Unsafe макросы
- Публичные unsafe методы

### Критичные модули (уделять больше внимания):
- `administration/` - аутентификация, пароли, ban
- `engine/network/` - сетевой ввод
- `engine/scripting/` - выполнение скриптов
- `engine/core/comm.cpp` - главный цикл
- `engine/db/` - работа с БД
- `gameplay/economics/` - деньги, предметы
- `gameplay/fight/` - боевая система

### Пропускать детальный анализ (быстрый просмотр):
- Константы (`*_constants.cpp`)
- Простые геттеры/сеттеры
- Очевидно безопасный код

## Оценка времени

- **Header файл**: ~30 сек
- **Простой .cpp**: ~1-2 мин
- **Сложный .cpp**: ~3-5 мин
- **Критичный модуль**: ~5-10 мин

**Всего**: ~30-50 часов чистого времени
**В сессии**: ~30-40 файлов (зависит от сложности)
